# Практическое занятие №14: WebSocket в FastAPI

---

## 1. Введение в WebSocket

### 1.1. Ограничения протокола HTTP

Протокол HTTP изначально проектировался по модели "запрос-ответ". Клиент отправляет запрос, сервер возвращает ответ, после чего соединение закрывается или переходит в режим ожидания нового запроса. Сервер не имеет возможности самостоятельно инициировать отправку данных клиенту без предварительного запроса с его стороны. Данную модель можно описать как "игру в одну калитку", где инициатива всегда принадлежит клиенту.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           HTTP: ОДНОНАПРАВЛЕННАЯ СВЯЗЬ                      │
│                                                                             │
│    ┌────────┐                                           ┌────────┐          │
│    │        │  ──────── Запрос (Request) ─────────────► │        │          │
│    │ Клиент │                                           │ Сервер │          │
│    │        │  ◄─────── Ответ (Response) ────────────── │        │          │
│    └────────┘                                           └────────┘          │
│                                                                             │
│    Клиент ВСЕГДА инициирует соединение.                                     │
│    Сервер ТОЛЬКО отвечает на запросы.                                       │
│    Сервер НЕ МОЖЕТ самостоятельно "позвонить" клиенту.                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.2. WebSocket как решение проблемы двунаправленной связи

WebSocket представляет собой протокол полнодуплексной связи поверх TCP. После установления соединения обе стороны могут отправлять сообщения в любой момент времени независимо друг от друга. Это можно описать как "игру в оба ворота", где и клиент, и сервер имеют равные права на инициацию обмена данными.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                       WEBSOCKET: ДВУНАПРАВЛЕННАЯ СВЯЗЬ                      │
│                                                                             │
│    ┌────────┐                                           ┌────────┐          │
│    │        │  ════════════════════════════════════════ │        │          │
│    │        │  ──────── Сообщение от клиента ────────►  │        │          │
│    │ Клиент │  ◄─────── Сообщение от сервера ─────────  │ Сервер │          │
│    │        │  ──────── Сообщение от клиента ────────►  │        │          │
│    │        │  ◄─────── Сообщение от сервера ─────────  │        │          │
│    │        │  ════════════════════════════════════════ │        │          │
│    └────────┘         Постоянное соединение             └────────┘          │
│                                                                             │
│    Обе стороны МОГУТ инициировать отправку сообщений.                       │
│    Соединение СОХРАНЯЕТСЯ на протяжении всей сессии.                        │
│    Сервер МОЖЕТ самостоятельно отправить данные клиенту.                    │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.3. Сравнение WebSocket с другими протоколами

Помимо WebSocket существуют и другие протоколы, обеспечивающие двустороннюю или асинхронную связь. Рассмотрим основные из них.

#### TCP (Transmission Control Protocol)

TCP является транспортным протоколом, обеспечивающим надёжную доставку данных с сохранением порядка пакетов. WebSocket работает поверх TCP, добавляя к нему семантику сообщений и веб-совместимость.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                    TCP                                      │
│                                                                             │
│    ┌────────┐                                           ┌────────┐          │
│    │        │  ═══════ Потоковое соединение ══════════  │        │          │
│    │ Клиент │  [байт][байт][байт][байт][байт]...        │ Сервер │          │
│    │        │  ══════════════════════════════════════   │        │          │
│    └────────┘                                           └────────┘          │
│                                                                             │
│    Преимущества:                                                            │
│      - Надёжная доставка с подтверждениями                                  │
│      - Сохранение порядка данных                                            │
│      - Контроль потока и перегрузок                                         │
│                                                                             │
│    Недостатки:                                                              │
│      - Потоковый протокол (нет понятия "сообщение")                         │
│      - Требует самостоятельного разбора границ данных                       │
│      - Не работает напрямую в браузере                                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### UDP (User Datagram Protocol)

UDP обеспечивает быструю передачу данных без гарантии доставки и порядка. Применяется в сценариях, где скорость важнее надёжности.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                    UDP                                      │
│                                                                             │
│    ┌────────┐                                           ┌────────┐          │
│    │        │  ─────── Датаграмма 1 ──────────────────► │        │          │
│    │        │  ─────── Датаграмма 2 ───────────X        │        │          │
│    │ Клиент │  ─────── Датаграмма 3 ──────────────────► │ Сервер │          │
│    │        │  ◄────── Датаграмма A ────────────────────│        │          │
│    │        │  ◄────── Датаграмма B ───────────X        │        │          │
│    └────────┘                                           └────────┘          │
│                                                                             │
│    Преимущества:                                                            │
│      - Минимальные задержки                                                 │
│      - Нет накладных расходов на установку соединения                       │
│      - Подходит для стриминга, игр, VoIP                                    │
│                                                                             │
│    Недостатки:                                                              │
│      - Нет гарантии доставки                                                │
│      - Нет гарантии порядка                                                 │
│      - Не работает напрямую в браузере                                      │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### Преимущества WebSocket

WebSocket объединяет преимущества обоих подходов и добавляет веб-совместимость.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                               WEBSOCKET                                     │
│                                                                             │
│    ┌────────┐    HTTP Upgrade (101)    ┌────────┐                           │
│    │        │  ══════════════════════  │        │                           │
│    │        │  [Сообщение 1]────────►  │        │                           │
│    │ Клиент │  ◄────────[Сообщение A]  │ Сервер │                           │
│    │        │  [Сообщение 2]────────►  │        │                           │
│    │        │  ◄────────[Сообщение B]  │        │                           │
│    └────────┘  ══════════════════════  └────────┘                           │
│                                                                             │
│    Преимущества:                                                            │
│      - Работает в браузере через JavaScript API                             │
│      - Надёжная доставка (поверх TCP)                                       │
│      - Чёткие границы сообщений (не поток байтов)                           │
│      - Совместимость с портами 80/443 и прокси                              │
│      - Низкие накладные расходы после установки соединения                  │
│      - Поддержка текстовых и бинарных данных                                │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2. Внутреннее устройство WebSocket

### 2.1. Процесс установки соединения (Handshake)

Соединение WebSocket начинается с HTTP-запроса, содержащего заголовок `Upgrade: websocket`. Сервер отвечает статусом 101 Switching Protocols, после чего протокол переключается на WebSocket.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          WEBSOCKET HANDSHAKE                                │
│                                                                             │
│    Клиент                                                    Сервер        │
│       │                                                         │           │
│       │   GET /ws HTTP/1.1                                      │           │
│       │   Host: example.com                                     │           │
│       │   Upgrade: websocket                                    │           │
│       │   Connection: Upgrade                                   │           │
│       │   Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==           │           │
│       │   Sec-WebSocket-Version: 13                             │           │
│       │ ──────────────────────────────────────────────────────► │           │
│       │                                                         │           │
│       │   HTTP/1.1 101 Switching Protocols                      │           │
│       │   Upgrade: websocket                                    │           │
│       │   Connection: Upgrade                                   │           │
│       │   Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=    │           │
│       │ ◄────────────────────────────────────────────────────── │           │
│       │                                                         │           │
│       │ ═══════════ WebSocket соединение установлено ═══════════│           │
│       │                                                         │           │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2. Механизм Ping/Pong

Для поддержания активности соединения и проверки его состояния в протоколе WebSocket предусмотрены контрольные кадры Ping и Pong. Сервер периодически отправляет Ping, клиент обязан ответить Pong. Это позволяет обнаружить разорванные соединения и предотвратить их закрытие промежуточными прокси-серверами.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           PING/PONG МЕХАНИЗМ                                │
│                                                                             │
│    Клиент                                                    Сервер        │
│       │                                                         │           │
│       │ ◄─────────────── PING (opcode 0x9) ──────────────────── │           │
│       │                                                         │           │
│       │ ────────────────  PONG (opcode 0xA) ──────────────────► │           │
│       │                                                         │           │
│       │                    ... время ...                        │           │
│       │                                                         │           │
│       │ ◄─────────────── PING (opcode 0x9) ──────────────────── │           │
│       │                                                         │           │
│       │ ────────────────  PONG (opcode 0xA) ──────────────────► │           │
│       │                                                         │           │
│                                                                             │
│    Интервал Ping обычно составляет 20-30 секунд.                            │
│    Если Pong не получен в течение таймаута - соединение закрывается.        │
│                                                                             │
│    В Uvicorn настраивается через:                                           │
│      --ws-ping-interval 20                                                  │
│      --ws-ping-timeout 10                                                   │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.3. Субпротоколы

При установке соединения клиент может предложить один или несколько субпротоколов через заголовок `Sec-WebSocket-Protocol`. Субпротокол определяет формат и семантику сообщений поверх WebSocket. Сервер выбирает один из предложенных субпротоколов и указывает его в ответе.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            СУБПРОТОКОЛЫ                                     │
│                                                                             │
│    Клиент                                                    Сервер        │
│       │                                                         │           │
│       │   Sec-WebSocket-Protocol: json, msgpack, protobuf       │           │
│       │ ──────────────────────────────────────────────────────► │           │
│       │                                                         │           │
│       │   Sec-WebSocket-Protocol: json                          │           │
│       │ ◄────────────────────────────────────────────────────── │           │
│       │                                                         │           │
│       │ ══════════════ Обмен JSON-сообщениями ══════════════════│           │
│       │                                                         │           │
│                                                                             │
│    Примеры субпротоколов:                                                   │
│      - json        : обмен данными в формате JSON                           │
│      - msgpack     : бинарный формат MessagePack                            │
│      - graphql-ws  : протокол GraphQL subscriptions                         │
│      - wamp        : Web Application Messaging Protocol                     │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.4. Типы кадров (Frames)

Данные в WebSocket передаются кадрами. Существуют кадры данных (текст, бинарные) и управляющие кадры (ping, pong, close).

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                             ТИПЫ КАДРОВ                                     │
│                                                                             │
│    Кадры данных:                                                            │
│      - Text   (opcode 0x1)  : текстовые данные UTF-8                        │
│      - Binary (opcode 0x2)  : бинарные данные                               │
│                                                                             │
│    Управляющие кадры:                                                       │
│      - Close  (opcode 0x8)  : закрытие соединения                           │
│      - Ping   (opcode 0x9)  : проверка активности                           │
│      - Pong   (opcode 0xA)  : ответ на ping                                 │
│                                                                             │
│    Структура кадра:                                                         │
│    ┌───────┬───────┬───────────────┬──────────────────────────────────┐     │
│    │ FIN   │Opcode │ Payload Len   │ Payload Data                     │     │
│    │ 1 bit │4 bits │ 7/16/64 bits  │ N bytes                          │     │
│    └───────┴───────┴───────────────┴──────────────────────────────────┘     │
│                                                                             │
│    Клиент ОБЯЗАН маскировать свои кадры (Masking Key).                      │
│    Сервер НЕ маскирует кадры.                                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 3. Реализация WebSocket в FastAPI

FastAPI использует Starlette для работы с WebSocket. Основной класс `WebSocket` предоставляет методы для приёма и отправки сообщений.

### 3.1. Базовые методы

Класс `WebSocket` предоставляет следующие основные методы:

- `accept()` - принять соединение после рукопожатия
- `close(code, reason)` - закрыть соединение с указанием кода и причины
- `send_text(data)` - отправить текстовое сообщение
- `send_bytes(data)` - отправить бинарные данные
- `send_json(data)` - отправить данные в формате JSON
- `receive_text()` - получить текстовое сообщение
- `receive_bytes()` - получить бинарные данные
- `receive_json()` - получить данные в формате JSON

### 3.2. Пример 1: Эхо-сервер и клиент

Смотри папку `example_1_echo`.

Эхо-сервер принимает сообщения от клиента и отправляет их обратно.

Запуск сервера:
```bash
uvicorn server:app --reload --port 8000
```

Запуск клиента (в отдельном терминале):
```bash
python client.py
```

### 3.3. Пример 2: Модульное тестирование

Смотри папку `example_2_testing`.

FastAPI предоставляет `TestClient` для тестирования WebSocket-эндпоинтов без запуска реального сервера.

Запуск тестов:
```bash
pytest test_websocket.py -v
```

## 4. Маршруты WebSocket с параметрами

### 4.1. Path-параметры

WebSocket-маршруты поддерживают path-параметры аналогично HTTP-маршрутам. Это позволяет создавать динамические эндпоинты, например, для комнат чата.

### 4.2. Query-параметры

Query-параметры передаются в URL при установке соединения. Они часто используются для передачи токенов авторизации или идентификаторов пользователей.

### 4.3. Body-параметры

В отличие от HTTP, WebSocket не поддерживает тело запроса при установке соединения. Все данные, которые требуется передать, должны передаваться либо через query-параметры, либо первым сообщением после установки соединения.

Смотри папку `example_3_params`.

## 5. Чат с множеством клиентов

### 5.1. ConnectionManager

Для управления множеством подключений используется паттерн ConnectionManager. Он хранит активные соединения и предоставляет методы для рассылки сообщений.

### 5.2. Событийная модель

Событийная модель организует обработку WebSocket через колбэки:

- `on_connect` - вызывается при подключении клиента
- `on_receive` - вызывается при получении сообщения
- `on_disconnect` - вызывается при отключении клиента

Данный подход близок к событийной модели Qt (сигналы и слоты) и упрощает организацию кода.

Смотри папку `example_4_chat`.

## 6. Передача файлов по WebSocket

### 6.1. Отправка небольших файлов

Небольшие файлы можно отправить одним бинарным сообщением через `send_bytes()`.

### 6.2. Чанкованная передача больших файлов

Большие файлы необходимо разбивать на части (чанки) и отправлять несколькими сообщениями. На принимающей стороне чанки собираются в исходный файл. Данный подход аналогичен работе QTcpSocket в Qt, где при превышении примерно 64 КБ данные автоматически разделяются на несколько пакетов.

Смотри папку `example_5_files`.

## 8. Дополнительно: Server-Sent Events (SSE)

### 8.1. Что такое SSE

Server-Sent Events представляет собой технологию однонаправленной передачи данных от сервера к клиенту через HTTP-соединение. В отличие от WebSocket, SSE использует обычный HTTP и не требует специального рукопожатия.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    SSE: ОДНОНАПРАВЛЕННЫЙ ПОТОК ОТ СЕРВЕРА                   │
│                                                                             │
│    ┌────────┐                                           ┌────────┐          │
│    │        │   GET /events HTTP/1.1                    │        │          │
│    │        │ ────────────────────────────────────────► │        │          │
│    │        │                                           │        │          │
│    │        │   HTTP/1.1 200 OK                         │        │          │
│    │        │   Content-Type: text/event-stream         │        │          │
│    │ Клиент │ ◄──────────────────────────────────────── │ Сервер │          │
│    │        │                                           │        │          │
│    │        │ ◄──────── data: событие 1                 │        │          │
│    │        │ ◄──────── data: событие 2                 │        │          │
│    │        │ ◄──────── data: событие 3                 │        │          │
│    │        │           ...                             │        │          │
│    └────────┘                                           └────────┘          │
│                                                                             │
│    Сервер отправляет данные, клиент только принимает.                       │
│    Формат: "data: содержимое\n\n"                                           │
│    Автоматическое переподключение при разрыве.                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 8.2. Когда использовать SSE вместо WebSocket

SSE предпочтительнее WebSocket в следующих случаях:

- Требуется только передача данных от сервера к клиенту (уведомления, ленты новостей, биржевые котировки)
- Не требуется передача данных от клиента к серверу в реальном времени
- Важна простота реализации и отладки
- Требуется автоматическое переподключение (встроено в EventSource API)

WebSocket предпочтительнее в случаях:

- Требуется двунаправленная связь в реальном времени
- Клиент активно отправляет данные серверу
- Требуется передача бинарных данных
- Критична минимальная задержка

## Коды закрытия WebSocket

| Код  | Название              | Описание                                        |
|------|-----------------------|-------------------------------------------------|
| 1000 | Normal Closure        | Нормальное завершение соединения                |
| 1001 | Going Away            | Сервер или клиент завершает работу              |
| 1002 | Protocol Error        | Ошибка протокола                                |
| 1003 | Unsupported Data      | Получены неподдерживаемые данные                |
| 1008 | Policy Violation      | Нарушение политики (например, авторизации)      |
| 1011 | Internal Error        | Внутренняя ошибка сервера                       |
